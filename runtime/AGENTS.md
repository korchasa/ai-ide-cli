# Runtime Module

- Responsibility: runtime abstraction for agent execution.
- Scope: adapter lookup, runtime option resolution, runtime capability metadata.
- Supported runtimes: `claude`, `opencode`, `cursor`, `codex`.
- Key decisions:
  - `claude` remains the default runtime for backward compatibility.
  - `runtime_args` is the universal extension point for all runtimes.
  - Engine-level HITL is runtime-agnostic at the engine layer: Claude uses `AskUserQuestion` permission denials; OpenCode uses an injected local MCP tool exposed per invocation through `OPENCODE_CONFIG_CONTENT`.
  - `opencode` continuation/resume uses `opencode run --session <id>`. `onToolUseObserved` fires once per non-HITL `tool_use` event whose `state.status` reaches a terminal value (`completed`/`failed`); tool id is taken from `part.id`, falling back to `part.callID`. Transcript is obtained post-run by invoking `opencode export <sessionId>` and writing stdout to a temp file (path surfaced as `CliRunOutput.transcript_path`); export failures are swallowed best-effort. Typed stream-event union `OpenCodeStreamEvent` is exported for consumers narrowing `RuntimeInvokeOptions.onEvent`.
  - `cursor` uses `cursor agent -p` headless mode with `--output-format stream-json`; session resume via `--resume <chatId>`; permissions bypass via `--yolo`. No HITL or transcript support — HITL unsupported by design (`capabilities.hitl = false`); see `cursor/AGENTS.md` for rationale. Stream-json events are typed via `cursor/stream.ts` (`CursorStreamEvent` discriminated union — system/user/thinking/assistant/tool_call/result, FR-L30). `onToolUseObserved` fires on every `tool_call/started` event (Cursor's tool calls are sibling top-level events with a `tool_call.<name>ToolCall.{args|result}` wrapper, NOT inline `tool_use` blocks like Claude); `unwrapCursorToolCall` flattens the wrapper into `{name, args?, result?, errorMessage?}`. `cursorHooks.onAssistant` provides a per-assistant-turn typed lifecycle hook.
  - `codex` uses `codex exec --experimental-json` headless mode; prompt is written to the child's **stdin**, not argv; session resume via positional `resume <threadId>`. Permission modes (`default`/`plan`/`acceptEdits`/`bypassPermissions`) map to combinations of `--sandbox` + `--config approval_policy=…`; Codex-native sandbox/approval values pass through unchanged. HITL via per-invocation `--config mcp_servers.hitl.command/args` registering a local stdio MCP server (shared with OpenCode in `hitl-mcp.ts`); `mcp_tool_call` items for `hitl.request_human_input` are intercepted and surfaced as `CliRunOutput.hitl_request`. Transcript path resolved post-run from `~/.codex/sessions/YYYY/MM/DD/rollout-*-<thread_id>.jsonl` into `CliRunOutput.transcript_path`. `launchInteractive` spawns the Codex TUI with skills copied into `~/.agents/skills/<name>/`. `onToolUseObserved` fires for `command_execution`/`file_change`/`mcp_tool_call`/`web_search` items. `openSession` switches to the **experimental** `codex app-server --listen stdio://` JSON-RPC transport (NOT `codex exec`) so callers can push follow-up user messages mid-turn via `turn/steer` — the `exec` transport closes stdin immediately after the initial prompt and therefore cannot accept follow-ups. See `codex/app-server.ts` (transport) and `codex/session.ts` (thread/turn semantics).
  - `fetchCapabilitiesSlow(opts)` is an optional capability (`capabilities.capabilityInventory: true`) that enumerates every skill and slash command the runtime currently exposes in a given `cwd`. Implementation is uniform across all four adapters: issue one LLM turn via `adapter.invoke` with a fixed system + task prompt (`CAPABILITY_INVENTORY_SYSTEM_PROMPT` / `CAPABILITY_INVENTORY_PROMPT`), then `parseCapabilityInventoryResponse` → `{runtime, skills, commands}`. Runtime-specific schema enforcement: Claude passes `--json-schema <inline>` + `--max-turns 1`; Codex writes `CAPABILITY_INVENTORY_SCHEMA` to a temp file and passes `--output-schema <path>`; OpenCode and Cursor have no schema flag and rely on the prompt alone (parser tolerates markdown-fenced or prose-embedded JSON). The method is **expensive** (full LLM turn per call, seconds-to-minutes, model-priced) — callers should cache results; hence the `Slow` suffix.
  - `openSession(opts)` is an optional capability (`capabilities.session: true`) for long-lived streaming-input sessions: caller may push additional user messages via `session.send(content)` while the subprocess is alive, consume normalized `RuntimeSessionEvent` objects via `session.events`, signal graceful shutdown via `endInput()`, or SIGTERM via `abort()`.
  - **Uniform `RuntimeSession` contract** (same for every runtime — verified by `runtime/session_contract_test.ts`):
    - `send(content)` resolves once the runtime has **accepted** the input. It never waits for turn completion. Transport/runtime errors during turn processing surface via `events` and `done`, not via the `send` promise. `send` rejects with a typed `SessionError` subclass: `SessionInputClosedError` (after `endInput`), `SessionAbortedError` (after `abort`), or `SessionDeliveryError` (HTTP non-2xx, broken stdin, JSON-RPC failure). Consumers branch on `instanceof` — message prefixes are not part of the contract.
    - `endInput()` is **signal-only**: flips the "no more input" flag and returns promptly. Full-shutdown observation lives on `await session.done`.
    - `abort(reason?)` is idempotent SIGTERM (or transport-equivalent).
    - `events` is a single-consumer async iterable; re-iteration throws.
    - `done` always resolves with `RuntimeSessionStatus` (never rejects).
    - `sessionId` is part of the neutral contract (string readonly; suitable as `resumeSessionId` on a later `openSession`). Populated **synchronously** before `openSession()` resolves for OpenCode/Cursor/Codex; for Claude the CLI allocates the id in the subprocess and emits it in the first `system/init` event — the neutral handle returns `""` until that event has been parsed, then updates in place (getter-backed).
    - `RuntimeSession` deliberately does NOT expose `pid` — it's a leaky implementation detail (Cursor has no long-lived backing process). Runtime-specific handles (`ClaudeSession`, `CursorSession`, `OpenCodeSession`, `CodexSession`) may expose `pid` / native id aliases (`chatId`, `threadId`) as their own fields.
  - **Synthetic turn-end event (`SYNTHETIC_TURN_END = "turn-end"`).** Every adapter injects exactly one synthetic event per completed turn, immediately **after** the runtime's native turn-terminator. `raw` carries the native payload for consumers that want the per-runtime detail (result subtype, cost, error fields). Sources:
    - Claude — emitted by `adaptRuntimeSession`/`adaptEventCallback` after `event.type === "result"`.
    - OpenCode — injected by the dispatcher in `opencode/session.ts` on busy→idle transition (edge-triggered, so a server emitting both `session.idle` and `session.status idle` still yields one turn-end).
    - Cursor — emitted after the per-turn subprocess's native `result`.
    - Codex — emitted in `codex/session.ts` after the `turn/completed` JSON-RPC notification.
    Honesty note: turn-end marks "runtime is ready for the next input", not "turn succeeded". Detecting failure still requires inspecting prior events or the `raw` payload per runtime.
  - **Out of scope for sessions (by design).** `RuntimeSessionOptions` deliberately omits `timeoutSeconds`/`maxRetries`/`retryDelaySeconds` (present on `RuntimeInvokeOptions`): a streaming session is caller-owned — implement per-turn timeouts and retries via `AbortSignal` + reopen with `resumeSessionId`. Mid-session model/permissionMode/extraArgs changes likewise require reopening (the flags are bound to the subprocess at spawn time).
  - **Typed tool filter (`allowedTools` / `disallowedTools`, FR-L24).** First-class fields on `RuntimeInvokeOptions` and `RuntimeSessionOptions` mapping to Claude's `--allowedTools` / `--disallowedTools` (comma-joined into two argv tokens). New `capabilities.toolFilter: boolean` — Claude `true`; OpenCode / Cursor / Codex `false`. Every adapter runs the shared `validateToolFilter` from `runtime/tool-filter.ts` (throws synchronously on mutual exclusion, empty array / empty strings, or `extraArgs` key collision with `--allowedTools` / `--allowed-tools` / `--disallowedTools` / `--disallowed-tools` / `--tools`), so YAML-driven consumers see uniform error messages regardless of runtime. Non-Claude adapters additionally emit one `console.warn` on first set-value call per process (`warnToolFilterOnce` + module-level latch; test reset via exported `_resetToolFilterWarning`). `CLAUDE_RESERVED_FLAGS` is **not** extended with the tool-filter keys — the legacy `extraArgs: {"--allowedTools": "..."}` path stays backwards compatible when the typed field is unset.
  - **Abstract reasoning effort (`reasoningEffort`, FR-L25).** Runtime-neutral 4-level enum (`"minimal" | "low" | "medium" | "high"`) on `RuntimeInvokeOptions` and `RuntimeSessionOptions`. New `capabilities.reasoningEffort: boolean` — Claude / Codex / OpenCode `true`; Cursor `false`. Per-adapter mapping: Claude → `--effort <value>` with `"minimal"` degraded to `"low"` plus one-time warn (`mapReasoningEffortToClaude` in `claude/process.ts`); Codex → `--config model_reasoning_effort="<value>"` on both `codex exec` (invoke) and `codex app-server` (session), 1:1 no warn; OpenCode → `--variant <value>` on invoke and `body.variant = <value>` on session's `POST /session/:id/prompt_async` with one-time warn (value is provider-specific); Cursor validates then warns once and ignores. Shared `validateReasoningEffort` in `runtime/reasoning-effort.ts` throws synchronously on out-of-enum values or `extraArgs` collision with `--effort` / `--variant`. Reserved-flag lists are **not** extended — legacy `extraArgs: {"--effort": …}` / `{"--variant": …}` without the typed field still works. Test resets: `_resetClaudeReasoningEffortWarning` (Claude), `_resetReasoningEffortWarning` (OpenCode / Cursor).
  - **Normalized content extraction (`extractSessionContent`, FR-L23).** Runtime-neutral helper in `runtime/content.ts` that turns a `RuntimeSessionEvent` into a `NormalizedContent[]` — the primitive every UI consumer should use instead of writing per-runtime `raw.*` branches. Envelope is unchanged; `raw` stays accessible. Pure, never throws on malformed payloads, returns `[]` for synthetic events and unknown types.
    - **Kinds**:
      - `{kind:"text", text, cumulative}` — streaming assistant text. `cumulative: true` (Claude/OpenCode/Cursor) means the consumer should replace its buffer; `false` (Codex) means append the delta.
      - `{kind:"tool", id, name, input?}` — tool/command invocation. Order preserved when multiple blocks live in one event.
      - `{kind:"final", text}` — complete assistant reply for the just-ended turn.
    - **Per-runtime source events**:
      - **Claude**: `assistant` event fans out `message.content[]` in order (`text` → text content, `tool_use` → tool content, `thinking` skipped); `result` event emits `final` (empty string included — consumer decides whether to render).
      - **Cursor** (FR-L30, forked from Claude): `assistant` event fans out `message.content[]` text blocks only (Cursor never inlines tool blocks); `tool_call` events with `subtype: "started"` emit `NormalizedToolContent` via `unwrapCursorToolCall`; `subtype: "completed"` skipped to mirror Claude's decision-time emission; `result` event emits `final`.
      - **Codex**: bound to app-server v2 **camelCase** types from `codex app-server generate-ts` — NOT to `codex/process.ts:codexItemToToolUseInfo` (which handles the parallel snake_case NDJSON from `codex exec`). `item/agentMessage/delta` → delta text; `item/completed` with `item.type === "agentMessage"` → final text (from `item.text` directly — not `content[]`). Tool kinds: `commandExecution` / `fileChange` / `webSearch` → `name = item.type`; `mcpToolCall` → `name = "<server>.<tool>"`; `dynamicToolCall` → `name = item.tool`. Input is the item payload minus `id` / `type`.
      - **OpenCode**: `message.part.updated` with `part.type === "text"` → cumulative text; with `part.type === "tool"` at terminal state (`completed` / `failed`), non-HITL (filters on `OPENCODE_HITL_MCP_TOOL_NAME`) → tool content (id falls back to `part.callID` when `part.id` missing). Mirrors the FR-L16 terminal-state dispatch rule.
    - **Timing asymmetry**: Claude / Cursor emit tool content at assistant-decision time (before execution — UI can show "📖 Read …" spinner). OpenCode / Codex emit at completion time (UI renders after the tool finishes). Matches the same dispatch points FR-L16 uses.
    - **Documented gaps**:
      - OpenCode has no native final-text event — consumers build `final` by keeping the last `cumulative:true` text they saw and flushing it on `SYNTHETIC_TURN_END`.
      - Claude `thinking` blocks and `user` events carrying tool results are deliberately skipped (reserved for future `kind:"reasoning"` / `kind:"tool-result"` variants; additive, no breaking change).
      - OpenCode may emit terminal-state `message.part.updated` more than once for the same tool id — stateless extractor forwards every occurrence; consumer dedupes by `id` if needed.
  - **Shared infrastructure** (no per-adapter copy-paste):
    - `runtime/event-queue.ts`: `SessionEventQueue<T>` backs every `session.events`.
    - `runtime/session-adapter.ts`: `adaptRuntimeSession` + `adaptEventCallback` translate runtime-specific sessions into the neutral handle.
    - `runtime/content.ts`: `extractSessionContent` + `NormalizedContent` union — runtime-neutral content extraction (FR-L23).
  - **Per-runtime transport:**
    - **`claude`** — real streaming-input transport (`claude/session.ts`: `claude -p --input-format stream-json --output-format stream-json --verbose` with piped stdin/stdout). One subprocess for the whole session. `send` returns after stdin flush; `endInput` closes stdin and returns.
    - **`opencode`** — `opencode/session.ts` spawns a dedicated `opencode serve --port <free> --hostname 127.0.0.1`, creates a session via `POST /session`, subscribes to `GET /event` SSE, forwards `send()` to `POST /session/:id/prompt_async` (returns on HTTP 204). `endInput()` is signal-only — it flips `inputClosed` and schedules a background task that waits for `session.idle` then SIGTERMs the server.
    - **`cursor`** — _faux_ session (`cursor/session.ts`): Cursor CLI has no `--input-format stream-json` equivalent, so `openCursorSession` obtains a chat ID via `cursor agent create-chat` (or accepts `resumeSessionId`), then spawns a fresh `cursor agent -p --resume <chatId> <message>` subprocess for every queued send. `send()` **enqueues and returns immediately** — does not wait for the subprocess to spawn or complete. Sends are serialized through a worker queue. A synthetic `{type:"system",subtype:"init",synthetic:true,session_id:<chatId>}` event is pushed at open time; per-turn subprocess failures emit a synthetic `{type:"error",subtype:"send_failed"}` event (not a rejected send). `pid` on `CursorSession` is a getter — reflects the currently active subprocess or `0` while idle; not exposed on the neutral `RuntimeSession`. Model selection is silently dropped (Cursor's `--resume` does not accept `--model`); `systemPrompt` is merged into the **first** user message of newly created chats. Stderr is concatenated across all subprocesses.
    - **`codex`** — `codex/session.ts` + `codex/app-server.ts`: spawns `codex app-server --listen stdio://` (experimental JSON-RPC transport, NOT `codex exec`), performs `initialize`/`initialized` handshake, then `thread/start` (or `thread/resume`). First `send` maps to `turn/start` (returns on RPC ack); subsequent sends during an active turn map to `turn/steer` with `expectedTurnId` set to `activeTurnId`. `activeTurnId` is updated in two places: synchronously from the `TurnStartResponse.turn.id` / `TurnSteerResponse.turnId` fields the moment `client.request()` resolves, and asynchronously from `turn/started` / `turn/completed` notifications (`turn/started` overwrites with the authoritative id; `turn/completed` clears it). The synchronous path closes the response-vs-notification race where two back-to-back `send()` calls would both route through `turn/start` if the notification had not yet been drained. `send` wraps JSON-RPC failures (`CodexAppServerError` under `cause`) in `SessionDeliveryError` so consumers catch one class across all four runtimes. `endInput()` calls `CodexAppServerClient.closeStdin()` (signal-only, does not await process exit). `close()` remains available for callers that want the legacy "EOF + await done" blocking shutdown. Wire shape mirrors only the fields actually present in the upstream-generated `v2/{ThreadStartParams,ThreadResumeParams,TurnStartParams,TurnSteerParams,UserInput}.ts`; the previously emitted `experimentalRawEvents` (orphaned — not in any upstream schema) and `persistExtendedHistory: false` (no-op duplicate of the default) have been dropped. Targets `codex-cli >= 0.121.0`.
    - Normalized events preserve the runtime's raw payload under `event.raw` for consumers that need typed access beyond the common `{runtime, type}` shape; callers MUST check the capability flag before invoking.

## Gotchas

- **Latent circular import — entry-point-sensitive.** `runtime/index.ts` ↔ `{claude,codex,cursor,opencode}-adapter.ts` ↔ `{runtime}/process.ts · session.ts` form a cycle through `expandExtraArgs` / `CLAUDE_RESERVED_FLAGS`. When `runtime/index.ts` is the entry (the normal case via `getRuntimeAdapter()`), evaluation order resolves fine. When any `*-adapter.ts` module is the **direct** entry (e.g. `import { claudeRuntimeAdapter } from "./claude-adapter.ts"` from a test or consumer), `runtime/index.ts` evaluates `ADAPTERS = { claude: claudeRuntimeAdapter, … }` before `claudeRuntimeAdapter` is bound → TDZ (`Cannot access 'claudeRuntimeAdapter' before initialization`). **Workaround in tests and consumers:** always import via `getRuntimeAdapter("claude")` from `./index.ts`, not the adapter module directly. **Proper fix (deferred):** extract `expandExtraArgs` + reserved-flag arrays to a cycle-free utility module.
- **Claude `system/init` event exposes a structured capability list.** The first NDJSON line of `claude -p <prompt> --max-turns 0 --output-format stream-json --verbose` is `{type:"system", subtype:"init", slash_commands:[], skills:[], agents:[], plugins:[{name,path,source}], mcp_servers:[], tools:[], model, session_id, …}`. Names only — descriptions are not exposed. Potential fast-path for `fetchCapabilitiesSlow` on Claude if the uniform LLM-prompt approach ever becomes a bottleneck; deliberately not taken today to keep cross-runtime symmetry (OpenCode/Codex/Cursor have no equivalent structured endpoint).
- **Single canonical home for stream-event types.** Per-runtime discriminated unions live in `<runtime>/stream.ts` (claude, cursor, opencode) or `<runtime>/events.ts` (codex). Companion modules (`<runtime>/session.ts`, `<runtime>/process.ts`) MUST re-export from the canonical module — never redeclare. When upgrading a loose alias (e.g. `Record<string, any>`) to a typed union, grep the runtime dir for the alias name BEFORE adding the typed module: a duplicated declaration that survives both barrels (`<runtime>/session.ts` and the new `<runtime>/stream.ts`) collides on re-export through `mod.ts`. See `cursor/session.ts:CursorStreamEvent` for the canonical re-export pattern (FR-L30).
- **Deno stdin-writer leak detection.** For any subprocess spawned with `stdin: "piped"`, the stdin `WritableStreamDefaultWriter` lifecycle is tracked independently of the process — killing the process (SIGTERM, external `AbortSignal`, or natural exit) does NOT close the writer from Deno's perspective. Tests fail with `A child process stdin was opened during the test, but not closed during the test`. **Convention:** every piped-stdin code path must explicitly close the writer (`writer.close()` on graceful end; fallback to `writer.abort()` when the pipe is already broken) in a `finally` / `done`-promise finalizer. See `forceCloseStdin()` in `claude/session.ts` for the canonical pattern.
